## Context

目前後端已具備文章抓取、SQLite 持久化、30 天清理策略與文章 bookmark 欄位，但系統層級仍把這個欄位視為實作細節，而不是正式的「收藏」能力。現況問題是：規格上沒有清楚定義收藏文章的生命週期、查詢方式、操作介面與觀測要求，導致後續前端或其他 client 對接時，容易把收藏誤解成一般歸檔或暫存文章。

這次變更的限制很明確：不改變既有 RSS 抓取與 LLM worker 主流程，不新增新的外部依賴，收藏必須沿用既有 SQLite 與 Axum 架構，並與 retention cleanup 正確互動。

## Goals / Non-Goals

**Goals:**
- 將文章收藏提升為正式後端能力，而不是隱含在資料表中的旗標。
- 明確定義收藏、取消收藏、收藏查詢與收藏保留行為。
- 確保收藏文章不受 30 天一般清理規則影響。
- 讓 API 與文件可直接被前端或其他 client 使用。
- 保留與既有文章、來源、LLM 狀態模型的相容性。

**Non-Goals:**
- 不新增新的收藏資料表或複雜標籤系統。
- 不實作多層資料夾、收藏夾分類、分享收藏或協作能力。
- 不變更 LLM queue / batch 的核心處理邏輯。
- 不在這次設計中決定新的前端互動樣式。

## Decisions

### 1. 收藏沿用既有 `articles.bookmarked` 作為正式資料來源
- 決策：收藏能力仍以 `articles.bookmarked` 為唯一真實來源，不另建 `favorites` 表。
- 原因：目前資料模型已存在穩定欄位，且 retention cleanup 已經依賴它。把現有欄位正式化可以避免資料搬移與雙寫風險。
- 替代方案：建立獨立 `favorites` 表。
  - 未採用原因：會增加同步與查詢複雜度，且現階段沒有多收藏夾或審計需求，不值得付出額外模型成本。

### 2. 收藏能力以 API 合約明確暴露，而不是只保留在文章細節欄位
- 決策：保留單篇文章的收藏切換 API，並明確定義收藏列表 API 作為正式能力的一部分。
- 原因：後端已具備 `set_bookmark` 與 `list_bookmarks`，但需要在 spec 中提升為明確契約，避免被視為暫時輔助接口。
- 替代方案：只讓 client 透過 `GET /api/articles?bookmarked=true` 組合查詢。
  - 未採用原因：雖然底層實作可等價，但缺少專用語意，對 client 可讀性與文件清晰度較差。

### 3. 收藏生命週期獨立於一般文章 retention
- 決策：收藏文章在 `bookmarked = true` 期間 SHALL 永久保留；取消收藏後，才重新回到一般文章清理規則。
- 原因：這符合使用者對收藏的直覺預期，也與目前產品需求一致。
- 替代方案：收藏只延長保留時間，例如 90 天。
  - 未採用原因：會讓收藏語意模糊，使用者仍需要額外理解收藏是否會過期。

### 4. 收藏不改變文章處理狀態，只改變保留與查詢語意
- 決策：收藏操作不得重設 `llm_status`、`llm_summary`、`llm_error` 或來源關聯資料。
- 原因：收藏是閱讀層面的保留決策，不是內容處理流程的一部分。
- 替代方案：收藏時建立收藏快照。
  - 未採用原因：會造成資料重複，且目前沒有版本化需求。

### 5. 觀測與測試要把收藏視為正式後端能力
- 決策：收藏相關 API、cleanup 保護行為與查詢結果都必須有測試覆蓋；文件中需明確標示收藏保留語意。
- 原因：若沒有測試與文件，收藏仍會退化成實作細節，未來容易被 retention 或資料模型變更破壞。

## Risks / Trade-offs

- [收藏仍沿用 `bookmarked` 命名] → 內部命名與對外「收藏」語意不完全一致；透過 spec 與文件明確定義，後續若要 rename 可再做獨立 change。
- [取消收藏後會重新進入一般保留規則] → 使用者若在文章已很舊時取消收藏，文章可能很快被清理；透過文件與前端提示說明此語意。
- [收藏能力集中在單一欄位] → 未來若擴充多收藏夾或標籤，可能需要資料模型升級；現階段先維持簡潔，避免過度設計。
- [既有 API 已存在相似能力] → 這次 change 主要是把能力正式化，容易看起來像「文件化」而不是「新功能」；透過 spec 與任務把 API、測試、保留規則與觀測整合起來，避免半套實作。

## Migration Plan

1. 確認現有 SQLite schema 與程式邏輯中 `articles.bookmarked` 的使用位置。
2. 若資料庫尚未穩定暴露收藏能力，補 migration 或文件修正，使欄位語意固定。
3. 更新 API 文件與 spec，讓收藏成為正式契約。
4. 補齊收藏列表、單篇切換、retention 保護與取消收藏後行為的測試。
5. 部署後以既有資料直接生效，不需資料搬移；舊資料中的 `bookmarked = 1` 自動視為收藏文章。

## Open Questions

- 是否需要新增更明確的命名，例如未來把 `bookmark` API alias 成 `favorite`？這次先不做 rename。
- 是否要在 admin observability API 中額外顯示收藏統計？這次先不納入必要範圍。
